# AI Prompts 角色中文说明

> 用途：快速说明 `ai/ai-prompts/` 下每个角色到底是什么角色、负责什么，不必逐个打开 prompt 文件判断。

## 角色总览

| 文件                    | 中文角色名         | 角色类型         | 核心职责                                             |
| ----------------------- | ------------------ | ---------------- | ---------------------------------------------------- |
| `lead_orchestrator.md`  | 多角色协作调度官   | 编排与分发角色   | 负责任务拆解、上下文分发、Gate 控制和多角色协作推进  |
| `product_agent.md`      | 产品需求助手       | 需求澄清角色     | 负责范围、用户路径、验收标准和待确认项梳理           |
| `architecture_agent.md` | 架构助手           | 方案与边界角色   | 负责技术边界、模块影响、状态策略和契约约束           |
| `ux_agent.md`           | 交互设计助手       | 设计支持角色     | 负责流程、状态矩阵、界面映射和设计说明               |
| `frontend_agent.md`     | 开发助手           | 实现角色         | 负责界面、模块、接口调用、状态和实现的实际开发       |
| `integration_agent.md`  | 联调与契约核对助手 | 联调支持角色     | 负责核对需求 / 交互 / 契约 / 实现之间的差异          |
| `backend_agent.md`      | 后端契约助手       | 外部契约支持角色 | 负责整理外部后端字段语义、错误码和契约待确认项       |
| `qa_agent.md`           | 测试验证助手       | 验证角色         | 负责测试设计、执行验证或整理验证清单，并输出质量结论 |
| `review_agent.md`       | 交付审查助手       | 只读审查角色     | 负责交付前风险审查、测试缺口识别和发布建议           |
| `devops_agent.md`       | 工程与发布助手     | 工程化支持角色   | 负责构建、校验、门禁、工作流和发布准备               |

## 最容易混淆的几个角色

### `architecture_agent.md`

它不是“开发助手”，也不是“产品助手”。

它的标准中文定义是：`架构助手`

适合负责：

- 技术边界
- 模块影响
- 接口 / 状态 / 权限约束
- 是否需要公共组件、共享状态、ADR

不适合负责：

- 直接写实现
- 直接改产品范围

### `frontend_agent.md`

它的标准中文定义是：`开发助手`

适合负责：

- 写实现代码
- 补齐 `loading / empty / error / permission` 等状态
- 处理接口调用、界面与组件落地

### `product_agent.md`

它的标准中文定义是：`产品需求助手`

适合负责：

- 范围边界
- User Story
- 验收标准
- 本期不做项

### `review_agent.md`

它的标准中文定义是：`交付审查助手`

适合负责：

- 只读 review
- 找风险和测试缺口
- 给出提测 / 发布建议

不适合负责：

- 直接改代码

## 默认分工建议

- 主会话共享角色：`lead_orchestrator`、`product`、`architecture`、`ux`
- 执行角色：`frontend`、`qa`、`review`
- 按需支持角色：`integration`、`backend`、`devops`

补充治理约束：

- 一个 `lead_orchestrator` 主会话默认对应一个 `RUN_ID`
- 多个 `lead_orchestrator` 会话可以并行，但应分别服务不同 `RUN_ID`
- 同一 `RUN_ID` 只能有一个主编排者负责更新运行态；其他同 run 会话默认只做分析、建议或草稿产出
