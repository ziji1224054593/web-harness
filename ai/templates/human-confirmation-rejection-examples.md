# Human Confirmation Rejection Examples / 人工驳回处理示例

> 用途：给人工确认节点提供可直接参考的驳回写法。  
> 原则：每次驳回都要写清楚 4 件事：
>
> 1. 驳回原因
> 2. 回退目标
> 3. 是否沿用当前 `RUN_ID`
> 4. 复审入口

## 1. 收敛确认驳回示例

### 示例 1：范围没收准，回到 Lead 收敛

- 当前阶段：收敛确认
- 结果：驳回并回退上游
- 驳回原因：
  - 本轮需求把“车贷首页状态展示”与“账单全流程”混在同一个 run 中，范围过大
  - 当前 run 的本期范围和范围外事项没有分清
- 回退目标：
  - `lead_orchestrator`
- 是否沿用当前 `RUN_ID`：
  - 否，先由 Lead 判断是否拆成两个 run
- 复审入口：
  - 回到“收敛确认”

可直接写法：

```text
驳回并回退上游。
原因：当前 run 同时混入首页状态展示与账单全流程，范围不清，已超出单轮交付边界。
回退目标：Lead Orchestrator。
RUN_ID 策略：暂不沿用，先判断是否拆分为新 run。
复审入口：收敛确认。
```

### 示例 2：仍属当前需求，但架构边界没讲清

- 当前阶段：收敛确认
- 结果：驳回并回退上游
- 驳回原因：
  - 是否新增 store、是否新增公共组件、接口字段来源都未明确
- 回退目标：
  - `architecture`
- 是否沿用当前 `RUN_ID`：
  - 是
- 复审入口：
  - 回到“收敛确认”

可直接写法：

```text
驳回并回退上游。
原因：当前仍属于本 run，但技术边界未收敛，store / 公共组件 / 字段来源不明确。
回退目标：Architecture。
RUN_ID 策略：沿用当前 RUN_ID。
复审入口：收敛确认。
```

## 2. Frontend 初版确认驳回示例

### 示例 1：只是实现问题，回给 Frontend

- 当前阶段：Frontend 初版确认
- 结果：驳回并返工
- 驳回原因：
  - 页面功能方向对，但缺少 `empty` 和 `permission` 状态
  - 部分文案没有接入国际化
- 回退目标：
  - `frontend`
- 是否沿用当前 `RUN_ID`：
  - 是
- 复审入口：
  - 回到“Frontend 初版确认”

可直接写法：

```text
驳回并返工。
原因：当前实现方向正确，但缺少 empty / permission 状态，且部分文案未接入 i18n。
回退目标：Frontend。
RUN_ID 策略：沿用当前 RUN_ID。
复审入口：Frontend 初版确认。
```

### 示例 2：不是实现问题，是前置理解偏了

- 当前阶段：Frontend 初版确认
- 结果：驳回并回退上游
- 驳回原因：
  - 当前页面增加了未在收敛阶段确认的操作入口
  - 问题不是“代码写错”，而是“本轮范围理解偏了”
- 回退目标：
  - `lead_orchestrator`
- 是否沿用当前 `RUN_ID`：
  - 是，先重新收敛；如范围变化过大再决定是否新开 run
- 复审入口：
  - 回到“收敛确认”

可直接写法：

```text
驳回并回退上游。
原因：当前实现加入了未在收敛阶段确认的功能入口，属于范围理解偏差，不是单纯实现问题。
回退目标：Lead Orchestrator。
RUN_ID 策略：暂沿用当前 RUN_ID，先重新收敛后再决定是否拆 run。
复审入口：收敛确认。
```

## 3. 交付前确认驳回示例

### 示例 1：实现可用，但验证不够，回给 QA

- 当前阶段：交付前确认
- 结果：驳回并返工
- 驳回原因：
  - QA 报告缺少权限态验证
  - 没有说明失败流和空态的覆盖情况
- 回退目标：
  - `qa`
- 是否沿用当前 `RUN_ID`：
  - 是
- 复审入口：
  - 回到“交付前确认”

可直接写法：

```text
驳回并返工。
原因：当前 QA 结论缺少权限态、失败流和空态的验证说明，证据不足以支持交付。
回退目标：QA。
RUN_ID 策略：沿用当前 RUN_ID。
复审入口：交付前确认。
```

### 示例 2：Review 发现风险过高，回给 Frontend

- 当前阶段：交付前确认
- 结果：驳回并返工
- 驳回原因：
  - Review 发现关键状态遗漏，存在明显回归风险
- 回退目标：
  - `frontend`
- 是否沿用当前 `RUN_ID`：
  - 是
- 复审入口：
  - 回到“交付前确认”

可直接写法：

```text
驳回并返工。
原因：Review 发现关键状态遗漏，当前结果存在明显回归风险，不适合直接交付。
回退目标：Frontend。
RUN_ID 策略：沿用当前 RUN_ID。
复审入口：交付前确认。
```

### 示例 3：已经不是当前交付目标，冻结当前 run

- 当前阶段：交付前确认
- 结果：驳回并回退上游
- 驳回原因：
  - 在交付前发现本轮已经混入另一个需求目标
  - 如果继续修，会导致当前 run 目标失焦
- 回退目标：
  - `lead_orchestrator`
- 是否沿用当前 `RUN_ID`：
  - 否，冻结当前 run，并新开 `RUN_ID`
- 复审入口：
  - 新 run 的“收敛确认”

可直接写法：

```text
驳回并回退上游。
原因：当前交付已混入新的独立需求目标，继续在本 run 内修复会导致交付边界失焦。
回退目标：Lead Orchestrator。
RUN_ID 策略：不沿用当前 RUN_ID，冻结当前 run，新开 RUN_ID。
复审入口：新 run 的收敛确认。
```

## 4. 最短写法模板

如果你平时只想写最短版本，可以直接套下面这个格式：

```text
结果：驳回并返工 / 驳回并回退上游
原因：
回退目标：
RUN_ID 策略：沿用当前 RUN_ID / 暂停当前 RUN_ID / 新开 RUN_ID
复审入口：收敛确认 / Frontend 初版确认 / 交付前确认
```

## 5. 使用建议

- 收敛阶段被驳回，优先回到 `Lead / Product / Architecture / UX`
- Frontend 初版被驳回，先判断是“实现问题”还是“收敛问题”
- 交付前被驳回，先判断是“代码问题”“验证问题”还是“方向问题”
- 除非已经偏离当前交付目标，否则优先沿用当前 `RUN_ID`
- 只有当问题已经不属于当前 run 目标时，才冻结当前 run 并新开 `RUN_ID`
