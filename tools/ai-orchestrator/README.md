# Harness Orchestrator (`@harness/orchestrator`)

`tools/ai-orchestrator/` is the execution package for the Harness AI delivery flow. It turns machine-readable tasks in `ai/runtime/` into a run state, evidence chain, and projected Markdown checklists and dashboards. The package is stack-agnostic: the only project-specific input is a small validation contract (`project.yaml`), so the same runtime drives a frontend repo, a backend service, or any other workspace.

## What Lives Here

### Source of truth

- `ai/runtime/tasks/`: task specs, dependency graph, current task status
- `ai/runtime/runs/`: run state snapshots, including gate states and current artifact pointers
- `ai/runtime/indexes/`: generated task index used for incremental task loading
- `ai/runtime/locks/`: run-level lock files used for CAS-style write protection
- `ai/runtime/definitions/`: role registry and pipeline registry
- `ai/runtime/definitions/project.yaml`: validation command contract for the active workspace
- `ai/runtime/schemas/`: JSON Schemas for runtime contracts
- `tools/ai-orchestrator/`: executable runtime, adapters, and projection code

### Projection targets

These files are derived artifacts and should be regenerated by the orchestrator instead of edited as the primary state:

- `ai/tasks/task-backlog.md`
- `ai/checklists/gate-status.md`
- `ai/reports/change-log.md`
- `ai/context/runs/<RUN_ID>/current-artifacts.md`
- `ai/runtime/projections/runs/<RUN_ID>/`
- `ai/tasks/runs/<RUN_ID>-task-backlog.md`
- `ai/checklists/runs/<RUN_ID>-gate-status.md`
- `ai/reports/runs/<RUN_ID>-change-log.md`

## Runtime Layout

```text
tools/ai-orchestrator/
├── commands/            # CLI entry points
├── runtime/             # planner / scheduler / dispatcher / adapters
├── core/                # shared TS contracts for projections and transitions
├── types/               # runtime-facing TypeScript contracts
└── package.json         # subpackage dependencies and scripts
```

## Quickstart

Scaffold the runtime directories and a sample task chain, then plan and inspect the run:

```bash
# 1) Create runtime dirs + a sample G1→G6 task chain (RUN-SAMPLE-001)
pnpm ai:init

# 2) Point the validation contract at your workspace, then validate it
#    (edit ai/runtime/definitions/project.yaml first)
pnpm ai:validate-project

# 3) Plan the sample run and open the dashboard
pnpm ai:plan -- --run RUN-SAMPLE-001
pnpm ai:overview
```

`ai:init` writes the runtime skeleton directories (each with a `.gitkeep`) and, when `ai/runtime/tasks/` is empty, a complete sample `requirement → scan → solution → implementation → validation → review` task chain spanning gates G1 through G6. Use `--bare` to skip the sample tasks, or `--force` to overwrite existing samples.

## Command Usage

The project root exposes the preferred proxy scripts. Use those first:

```bash
pnpm ai:init
pnpm ai:plan -- --run <RUN_ID>
pnpm ai:run -- --run <RUN_ID>
pnpm ai:sync -- --run <RUN_ID>
pnpm ai:report -- --run <RUN_ID>
pnpm ai:artifact -- --run <RUN_ID> --key current_handoff --path ai/reports/handoff/<RUN_ID>-handoff.md
pnpm ai:handoff -- --run <RUN_ID> --by lead-a --to lead-b --reason "handoff after convergence"
pnpm ai:pause -- --run <RUN_ID> --by lead-a --reason "waiting for human confirmation"
pnpm ai:resume -- --run <RUN_ID> --by lead-a --reason "confirmation completed"
pnpm ai:retry -- --run <RUN_ID> --by lead-a --reason "retry after fix"
pnpm ai:validate-project
```

You can still run the subpackage directly when debugging the orchestrator package itself:

```bash
pnpm --dir tools/ai-orchestrator run ai:run -- --run <RUN_ID>
```

### Command responsibilities

- `ai:init`: scaffold the runtime directory skeleton and an optional sample G1→G6 task chain
- `ai:plan`: load tasks, create or refresh run state, and project the current backlog / gates / report
- `ai:run`: claim ready tasks with execution leases, execute them, commit results with lock + CAS, then refresh projections
- `ai:sync`: re-project JSON and Markdown artifacts without executing tasks; unchanged projections are skipped
- `ai:report`: print the minimal machine-readable run summary to stdout
- `ai:validate-project`: validate the active project contract before onboarding or automation
- `ai:approve` / `ai:reject`: record a human gate decision on a run
- `ai:artifact`: update current artifact pointers or superseded references on the run source-of-truth, then re-project
- `ai:handoff`: formally transfer or take over the primary orchestrator ownership for a run, then re-project
- `ai:post` / `ai:board`: post a message to and read the per-run coordination board
- `ai:overview`: render the cross-run dashboard
- `ai:pause` / `ai:resume`: pause or resume execution on a run
- `ai:retry`: move retryable `blocked` / `failed` tasks back to `ready`
- `ai:recover`: recover a run whose leases or locks were left in an inconsistent state

## Execution Model

`commands/run.mjs` follows this order:

1. Acquire the run lock and load the latest indexed task closure
2. Recover expired leases, unlock newly ready tasks, and claim up to the worker limit
3. Persist claimed task leases plus the next run revision through lock + CAS
4. Release the run lock and dispatch claimed tasks to execution adapters
5. Persist structured evidence under `ai/runtime/evidence/<runId>/`
6. Re-acquire the run lock, commit each task result, and unlock downstream tasks
7. Recompute incremental projections and only rewrite changed artifacts

### Adapter routing

- `validation` tasks are executed locally through `runtime/validation-adapter.mjs`, which runs the real workspace commands
- All other tasks are executed through a tool-capable coding agent via the Cursor SDK adapter

This split is intentional: requirement / scan / solution / implementation / review tasks are AI-authored, while repository checks must run the real workspace scripts rather than rely on an LLM summary.

## Validation Adapter

The validation adapter is the bridge between the project's own commands and the orchestrator evidence chain.

### Project contract driven checks

For every `validation` task, the adapter resolves its suite from `ai/runtime/definitions/project.yaml` instead of assuming any fixed stack.

The contract defines:

- `workspaceRoot`: the project root relative to the harness repository
- `validation.defaultSuite`: checks used when the task does not override them
- `validation.checks[]`: executable checks, aliases, and optional `runFrom` overrides

The example below uses generic lint/typecheck commands — replace the `command` values with whatever your project actually runs (a test runner, a build, a custom script, etc.):

```yaml
validation:
  defaultSuite:
    - lint
    - typecheck
  checks:
    - checkId: lint
      description: Lint check (replace with your project's command)
      command: pnpm lint
      runFrom: .
    - checkId: typecheck
      description: Type / build check (replace with your project's command)
      command: pnpm typecheck
      runFrom: .
```

### Optional task-level override

If a task needs a narrower suite, set `metadata.validationSuite` in the task YAML as a comma-separated list of project-defined `checkId` values or aliases:

```yaml
metadata:
  validationSuite: lint,typecheck
```

Supported entries are:

- any `checkId` declared by the project contract
- any alias declared for a validation check

### Result semantics

- If all checks pass, the task becomes `done`
- If any check exits non-zero, the task becomes `failed`
- If the suite is invalid or cannot be resolved, the task becomes `blocked`

### Evidence chain

Each validation check emits one structured evidence record. Persisted evidence contains:

- task and run identifiers
- the high-level task result summary
- the specific validation check result
- command, exit code, duration, and output snippets

Because gate projection already relies on `task.status + evidenceRefs`, G5 advances naturally:

- passed validation task with evidence -> `G5: passed`
- failed or blocked validation task -> `G5: blocked`

The gate projection also surfaces the latest validation summary in `ai/checklists/gate-status.md`, so the projected checklist is readable without opening the raw evidence JSON first.

## Environment Notes

### Coding agent tasks

- `@cursor/sdk` must be installed in `tools/ai-orchestrator/package.json`
- `CURSOR_API_KEY` must be available for agent-backed tasks
- `CURSOR_AGENT_MODEL` can override the default model

If the agent adapter cannot start, the task is marked `blocked` with a structured reason instead of being treated as success.

### Validation tasks

Validation tasks do not depend on the Cursor SDK. They only require the project contract's local commands to be runnable from the configured workspace.

## Project Contract

`ai/runtime/definitions/project.yaml` is a small validation contract for the active workspace. It does not define routing, preview automation, or delivery rules. Its current scope is intentionally narrow:

- validation suites and aliases
- workspace root for command execution

If the file is absent, the orchestrator falls back to a built-in default contract so existing flows remain runnable.

Run this check before onboarding a new repository:

```bash
pnpm --dir tools/ai-orchestrator run ai:validate-project
```

The command exits non-zero when the contract is not safe to use and reports:

- missing workspace or `runFrom` directories
- duplicate validation check IDs or alias conflicts
- unknown `validation.defaultSuite` entries

> Note: where a task or contract needs an API/contract source of truth, point it at the project's API/contract source (`project.yaml: contractSource`) rather than any hard-coded spec location.

## Typical Workflow

1. Define or update tasks under `ai/runtime/tasks/` (or start from `pnpm ai:init`)
2. Run `pnpm ai:plan -- --run <run-id>`
3. Use `pnpm ai:artifact -- --run <run-id> --key current_handoff --path <file>` when current artifact pointers change
4. Use `pnpm ai:handoff -- --run <run-id> --by <current-owner> --to <next-owner>` when the primary orchestrator changes
5. Run one or more `pnpm ai:run -- --run <run-id> --by <worker-id>` workers on the same run when you want coordinated parallel execution
6. Inspect:
   - `ai/runtime/runs/<run-id>.json`
   - `ai/runtime/evidence/<run-id>/`
   - `ai/runtime/projections/runs/<run-id>/`
   - `ai/checklists/gate-status.md`
   - `ai/context/runs/<run-id>/current-artifacts.md`
   - `ai/reports/change-log.md`
7. If a task is blocked or failed, fix the cause and run `pnpm ai:retry -- --run <run-id>`

## Design Rule

If a file answers "what is the contract or source-of-truth model?", it belongs in `ai/runtime/` or `types/`.

If a file answers "how is work executed against the real repo?", it belongs in `tools/ai-orchestrator/runtime/` or `commands/`.
