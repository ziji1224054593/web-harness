把一个需求,拆给多个 Agent
按阶段接力,不合格就打回返工

Harness 让六个角色身份按门禁接力推进一条需求:每道门禁人工确认放行,验收不过自动打回上一棒返工,返工—复验循环到通过;轮次触顶则熔断、转人工裁决。事实源单一、单写者写保护、多需求并行。

工作原理

六阶段主线 + 打回返工循环

每道门禁(Gn)都是一次人工确认。light 档可裁掉交互与审查两道门,小改动不必走全流程。

G1

需求提炼

product

把诉求收敛成范围、用户路径、验收标准与待确认项。

G2

模块扫描

architecture

定技术边界、目录影响、契约约束与组件复用策略。

G3

交互对齐

ux

产出页面地图、状态矩阵、组件映射;登记交互稿。

light 档可跳过
G4

前端实现

frontend

落地页面/组件/接口/状态/文案,补齐 loading、空、错、权限态。

G5

校验

qa

按真实链路 + 契约一致性验证,运行时取证,出质量结论。

G6

交付前审查

review

只读风险审查,识别回归与验证缺口,给发布建议。

light 档可跳过
1

前端交接 ai:post --kind done

实现完成产出 handoff,人工确认后登记交接物,测试才进场。

2

测试验证

按真实链路 + 字段契约逐项核验,采集请求/控制台证据。

↩

不过 → 打回 ai:post --kind reject

带编号问题清单弹回前端;reworkRounds += 1,只修清单内项,不扩范围。

3

复验 → 回到第 2 步

修复后重新交接复测,通过则放行 G5 进审查;仍不过再返工一轮。

✓

通过 ai:post --kind pass

门禁 G5 放行,交给审查身份做交付前的只读检查。

返工熔断 · circuit breaker
轮 1→轮 2→轮 3→熔断

每次 ai:retry 计一轮返工。轮次达 maxRounds(默认 3),熔断器跳闸:拒绝再次回流,升级人工裁决——避免前端↔测试之间无限打转。

状态持久化在 run 的 reworkState,熔断由状态机强制,不靠口头约定。

执行模型 · 任务怎么真正被执行

不是"会用工具的 Agent",是按类型分流的编排器

Harness 本身是确定性状态机,不跑 LLM 工具循环。它把每个任务按类型路由到对应执行器:能确定性验证的就跑真实命令,需要创作的才交给具备工具能力的编码 Agent;两条道都受角色门禁约束。

🧩

适配器路由

调度器按 task.type 分流:validation 走本地校验适配器,其余(需求/扫描/方案/实现/审查)走 Agent 适配器。同一条流水线里两种执行器并存。

🛠️

校验 = 跑真实命令

校验任务直接在工作区 exec 仓库脚本(如 lint / typecheck),凭退出码判定 done / failed / blocked,每条命令产一条结构化证据——绝不让 LLM "口头说通过"。

🤖

AI 任务 = 工具型编码 Agent

创作类任务委托给具备读写文件、跑命令能力的编码 Agent(Cursor SDK / composer)在真实仓库根目录执行;harness 只负责注入角色 prompt、任务边界与输出契约。

⛔

工具级门禁

每个角色声明 allowedTaskTypes;领到不被允许的任务类型直接 blocked。无 API Key / SDK 缺失 / 角色未注册,都返回结构化阻断,不会被当成成功。

没有"skill"概念:能力来自真实命令执行 + 委托的工具型 Agent。该用确定性脚本的地方就不让模型猜,该创作的地方才上 Agent。

核心特性

为多 Agent 协作而设计

把"谁在做、做到哪一棒、过没过门禁"变成可强制、可追溯的运行态,而不是一段口头约定。

🚦

阶段门禁

六个角色按阶段接力,每道门禁(G1–G6)需人工确认才放行,关键交接不被悄悄跳过。

↩️

打回—返工熔断

验收不过带问题清单打回上一棒,reworkRounds 计数;触顶 maxRounds 自动熔断,转人工裁决。

📥

拉取式执行

任务进就绪队列由 worker 轮询领取并上 lease 租约;依赖未完成不进队列,既不空转也不拿半成品开测。

🔒

单写者写保护

同一 RUN 单写者:CAS 版本号 + 目录锁,外来写以 revision mismatch 被拒,杜绝并发覆盖。

🧵

多 RUN 并行

每条 RUN 独立运行态 / 锁 / owner,跨 RUN 全并行;改同一代码区前需显式登记冲突并协调。

🔗

契约预检

进实现前对齐 业务字段 ↔ 接口契约 ↔ 前端类型 ↔ 渲染/提交,任一环对不上记为 contract gap 并阻断,不许相似字段顶替。

🎚️

分档裁剪

light / standard / full 按变更规模裁剪门禁链,小改动走轻量路径,不一刀切。

💬

聊天通道播报

每次交接 = 一条机器人消息,投影到 IM(Slack / 飞书 / 钉钉等)与 channel.md,全员可见、可追溯。

📄

单一事实源

事实源是 runs/<RUN>.json;台账、门禁、群对话都是它的投影,改派生件不影响真值。

快速上手

几条命令驱动一条 RUN

定义任务后,用代理脚本规划、执行、再投影。无需后端服务,运行态都落在仓库文件里。

# 安装依赖
pnpm install

# 1) 规划:载入任务、建/刷新运行态,投影台账 + 门禁 + 报告
pnpm ai:plan  -- --run RUN-20260623-001

# 2) 执行:领取就绪任务、执行、提交证据(lock + CAS)、刷新投影
pnpm ai:run   -- --run RUN-20260623-001

# 3) 仅重新投影(不执行),未变更的产物自动跳过
pnpm ai:sync  -- --run RUN-20260623-001

# 不过就打回,修复后把可重试任务移回就绪
pnpm ai:retry -- --run RUN-20260623-001 --reason "fixed"

# 打印机器可读运行摘要
pnpm ai:report -- --run RUN-20260623-001
